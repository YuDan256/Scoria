# Scoria Git 提交规范 (COMMIT_CONVENTION.md)

本文档规定了 Scoria 编译器的 Git 提交格式规范，旨在维护清晰的项目演进路线，降低 C 语言环境下的内存与 Bug 溯源成本，并强制要求在提交记录中留存编译原理推演与系统架构调整的设计逻辑。

## 1. 提交信息结构

每次提交必须严格遵循以下文本格式：

```
<type>(<scope>): <subject>
<空一行>
【动机】
（说明修改的缺陷背景、编译性能瓶颈或功能引入背景）
<空一行>
【原理】
（说明编译器状态机、AST 内存布局、IR 生成逻辑或类型系统校验的具体技术实现）
```

## 2. `<type>` 修改类型

必须且仅能使用以下类型标识：

*   feat     : 编译器基础结构扩充（新增词法规则、解析器语法支持或汇编指令扩展）。
*   fix      : 缺陷修复（修复语法树解析越界、内存池泄漏或错误的推导类型）。
*   perf     : 性能优化（优化 Arena 内存分配频率、降低哈希碰撞、消除冗余 IR 指令等）。
*   refactor : 内部重构（不增加新功能、不修复缺陷，仅进行 C 语言源码级的结构整理与解耦）。
*   core     : 核心规范变动（专用于修改最高级别的类型转换规则、内存别名规则等语言层面的法典行为）。
*   docs     : 文档编撰（如 README 更新或架构图说明）。
*   test     : 测试用例集成（新增 `.sco` 测试源文件，或测试断言的修改）。
*   chore    : 工程杂项（如 CMakeLists 构建脚本更改、调试器配置更新）。

## 3. `<scope>` 作用模块

界定当前提交影响的代码边界，必须与 Scoria 编译器物理目录划分严格对应：

*   frontend : 词法分析 (Lexer)、抽象语法树 (AST) 构建、递归下降语法解析 (Parser)。
*   middleend: 符号表与作用域控制、静态类型检查器 (Type Checker)。
*   ir       : 中间语言 (SIR) 定义、AST 到三地址码的平坦化生成引擎。
*   backend  : 寄存器分配器、特定平台汇编代码生成与目标二进制链接。
*   utils    : 编译器基础设施（内存池 Arena、字符串驻留池 StringPool、日志系统）。
*   driver   : 编译器入口参数解析与编译流水线生命周期控制。
*   stdlib   : 语言内置标准库（如 `systema.sco`）。
*   build    : 构建系统层面（CMakeLists 等非 C 语言源码文件）。

## 4. `<subject>` 标题要求

1. 长度限制在 50 个字符以内。
2. **强制使用拉丁语（Latin）**进行首行概括。
3. 强制以拉丁语动词祈使句开头，首字母小写（例如：`adde` [增加]、`emenda` [修复]、`perfice` [优化/完成]、`muta` [修改]、`remove` [移除]）。
4. 结尾切勿添加任何标点符号。

示例：`core(middleend): coge terminum inter litteram et p8` (强制 littera 与 p8 之间的边界)

## 5. `<正文>` 详细说明

在底层 C 语言开发中，所有涉及内存布局（如 `struct/union` 成员修改）或类型规则体系变更的提交，正文必须具备以下技术要素：

*   【动机】：阐明修改原因。例如 C 标准库 `malloc` 导致的内存碎片、AST 嵌套过深导致的遍历栈溢出、或用户通过特定语法绕过了系统类型检查。
*   【原理】：阐明底层物理实现。如讲解所采用的连续内存推移算法、AST 标记联合体（Tagged Union）的重构方案、或者异或指令（XOR）在 IR 层的常量折叠推导。文字均使用中文描述。

## 6. 标准提交示例

### 示例 1：利用 Arena 内存池替换 malloc
```
perf(utils): adde arenam pro nodis ast

【动机】
原先的解析器在处理包含大量符号的代码文件时，对每一个 ASTNode 进行独立的 malloc() 调用，导致跨页内存碎片严重，且在编译中断时易发生内存泄漏。

【原理】
1. 引入 memory_arena.c，开辟起始大小为 64MB 的连续虚拟内存。
2. 将 AST 节点的创建改为单向指针推进（bump-pointer），实现 O(1) 复杂度的分配。
3. 弃用对 AST 树的递归 free() 机制，在 driver 生命周期结束时一次性截断并释放整块内存。
```

### 示例 2：严格约束字符与数字的类型隔离
```
core(middleend): coge terminum strictum inter litteram et p8

【动机】
过去的类型系统继承了 C 语言的宽容度，允许 littera (字符型) 隐式赋予未受 muta 重铸的 p8 (无符号 8 位整型)，这打破了 Scoria 绝对的强类型安全设计。

【原理】
1. 在 type_checker.c 中加入强校验分支，从符号层隔离字符集与数值标识。
2. 凡是在 AST_ASSIGN 赋值逻辑两侧，侦测到物理等长但语义异构的组合，直接触发异常日志截断编译。
3. 仅当解析到外层包裹的 AST_MUTA (Cast) 节点时，才允许放行至 IR 生成阶段。
```

## 7. 提交模板配置

为保证提交格式统一，建议在 Scoria 根目录配置终端提交模板。请在项目根目录创建名为 `.gitmessage` 的文件，填入以下框架：

```
<type>(<scope>): <subject_in_latin>

【动机】


【原理】

```

随后，在终端执行以下命令，将其设为 Git 默认模版：

```bash
git config commit.template ./.gitmessage
```
